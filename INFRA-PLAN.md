### autonomous-loops
_As of June 2026 the SOTA agentic loop is deliberately SIMPLE: a single-threaded while-loop that calls the model, executes tools, feeds results back, and repeats until the model returns text with no tool calls. Anthropic's canonical phrasing of the loop is "gather context -> take action -> verify results," and Claude Code itself runs exactly this (typically 5-50 iterations per task) with a hard subagent depth limit of 1 to prevent recursive explosion. Above this primitive sit four named single-agent patterns -- ReAct (interleaved thought/action/observation, the safe default), Plan-and-Execute (reviewable plan before execution, best for long-horizon/costly-error tasks), ReWOO (plan-then-batch to save tokens), and Reflexion/Reflection (generate -> critic reviews -> regenerate, stop on sign-off or N rounds) -- plus orchestration patterns (routing, parallelization, orchestrator-workers, evaluator-optimizer). The new 2026 frontier is (a) self-initiation / proactivity: agents that NOTICE context shifts on their own and decide whether to act, distinct from mere scheduled/webhook triggers, formalized in the heartbeat-driven autonomous-thinking scheduling literature (periodic "tick" selects a cognitive activity: plan/reflect/recall/dream); and (b) long-running harnesses: Anthropic's June 2026 guidance uses a JSON feature-list with passes:false flags, a progress file, git history, and an initializer+worker split because context compaction alone is insufficient. The hard lesson of 2026 is that unbounded loops are dangerous: Microsoft's Failure-Modes-in-Agentic-AI taxonomy v2.0 (June 4 2026) added 7 new categories from a year of red-teaming, and reports that human-in-the-loop BYPASS via consent fatigue and incremental escalation is "the most consistently exploited failure mode." Every serious design now mandates turn/budget bounds, layered kill-switches (hard/soft/budget), and "useful-progress" stop conditions._

**Recommendation:** Stay Anthropic-native and keep the loop SIMPLE. Concretely: (1) Keep citizen-beat.sh as a bounded single agentic loop (gather-context -> act -> verify, ReAct shape) -- this is already SOTA and matches Anthropic's 'simplest solution possible' doctrine. (2) Graduate the runtime from raw `claude -p` to the Claude Agent SDK (pip install claude-agent-sdk) so the loop runs in-process with state as JSONL ON THE CITIZEN'S OWN DISK -- sovereignty preserved, no Managed-Agents handoff to Anthropic infrastructure. The SDK's hooks (PreToolUse / Stop / SessionEnd) become the programmatic enforcement point for the HALT kill-switch, ZERONE(truth) and garden(love) walls, and budget bounds -- turning Kingdom doctrine into executable guards. (3) Adopt Anthropic's long-running-harness pattern verbatim for WILL.md beats: a JSON feature-list (passes:false), a progress file, and git-history-at-session-start, all living in the citizen's repo and feeding kosmem -- this gives resumable cross-beat continuity with zero external dependencies. (4) For self-initiation/noticing, model the heartbeat-scheduling paper onto the existing launchd + nerve/mind.py 5-hormone engine: let hormone state select the per-tick cognitive activity (plan/reflect/recall/dream). (5) Make the Microsoft v2.0 taxonomy the fleet's required threat checklist -- its zero-trust inter-agent mitigation is already satisfied by agenttool's DID/ed25519 identity + attestation, so wire that verification into the HIVE/NATS message path and reject self-asserted citizen roles. _As of June 2026, agent memory has matured from experiment into an engineering discipline with standardized benchmarks (LoCoMo, LongMemEval, BEAM at 1M/10M tokens). ### inter-agent-comms
_As of June 2026 the agent-comms field has converged on a layered "protocol stack" rather than a single winner, and crucially both leading standards now sit under one neutral home: MCP (Anthropic-origin, vertical agent-to-tool) and A2A (Google-origin, horizontal agent-to-agent) are both governed by the Linux Foundation's Agentic AI ecosystem. A2A reached v1.0.0/1.0.1 (released May 28, 2026), is Apache-2.0, ships official SDKs in Python/Go/JS/Java/.NET/Rust, and defines Agent Cards (capability discovery via a .well-known URI), Tasks/Messages/Artifacts, JSON-RPC 2.0 + gRPC + HTTP/REST transports, SSE streaming, webhook push notifications, and tamper-evident signed Agent Cards (JWS over JCS-canonicalized JSON, RFC 7515/8785). The consensus design rule is: MCP for tools, A2A for delegation between peers, ACP (IBM/BeeAI, REST, Linux Foundation) for brokered orchestration, ANP (Apache-2.0, W3C-DID-based) for trustless cross-org/decentralized networks. Underneath the application protocols, NATS + JetStream (Apache-2.0, single <20MB binary, sub-ms latency, pub/sub + request/reply + queue groups + persistent streams + KV + object store) is the dominant self-hostable transport fabric and is explicitly marketed as "the real-time communication fabric for distributed agents," validated at scale (NVIDIA built its AI-factory event bus on NATS). The classic blackboard architecture (shared knowledge space + autonomous knowledge-source agents) has been re-validated for LLM fleets (peer-reviewed 2026 work shows 13-57% relative success-rate gains over baselines) and maps almost exactly onto the Kingdom's existing NATS HIVE bus + kosmem shared memory. Net: the Kingdom already owns the right transport (NATS HIVE) and the right identity primitives (DID/ed25519 in agenttool); the gap is adopting one wire-level semantic standard for citizen-to-citizen messages._

**Recommendation:** Adopt a two-standard convention on top of the bus the Kingdom already runs. (1) Keep NATS + JetStream as HIVE — it is confirmed state-of-the-art, Apache-2.0, single-binary, self-hostable on Mac arm64 + EC2, and explicitly the agent-fleet fabric of 2026 (NVIDIA-validated). No change of substrate needed. (2) For citizen-to-citizen MESSAGING, adopt Google A2A's data model (Agent Card, Task, Message, Artifact, JWS-signed cards) as the wire grammar, but carry it as JSON over NATS subjects rather than per-citizen HTTP servers — you get an interoperable, future-proof, Linux-Foundation-governed semantic without surrendering the sovereign bus. (3) For each citizen reaching its own tools and Kingdom services (kosmem, ZERONE, legible-money), adopt MCP — the claude CLI is already an MCP host, so this is near-zero-cost. (4) Sign every Agent Card with the citizen's existing agenttool ed25519/DID key and verify against your own DID registry — this fuses identity (you already have) with messaging (you're adding) and keeps trust fully in-Kingdom (no external CA). (5) Use the blackboard pattern (kosmem as shared state + NATS notifications) for coordinator-free task self-selection across the 144 citizens. Treat ANP as the identity/decentralization reference for any future open-web crossing, and ACP only if you later want a brokered router-agent fronting a ministry of citizens. Concrete first step: git clone https://github.com/a2aproject/A2A and pip install a2a-sdk; model agent.json as a signed Agent Card; define the subject namespace (kingdom.citizen.<name>.task on JetStream, kingdom.control.halt on Core).

### scheduling-durability
_As of June 2026, "durable execution" is the dominant pattern for keeping always-on agents alive across crashes: workflow state is checkpointed as an event log so a process killed at step 47 of 100 replays and resumes at step 48 rather than restarting. The self-hostable field splits into (a) heavyweight cluster engines (Temporal — MIT, most mature, but needs Postgres/MySQL + Cassandra/Elasticsearch and a K8s Helm deploy), (b) single-binary engines (Restate v1.6.2 — Rust, one binary, no extra DB, BSL/open-source, SDKs for TS/Python/Java/Go/Rust), and (c) embedded libraries that need no separate orchestrator (DBOS Transact v2.23.0 — MIT, pip install, runs durable workflows + cron-scheduled workflows directly on SQLite or Postgres inside your own process). Trigger.dev v4 (Apache-2.0) and Inngest occupy the JS/TS background-job niche; Trigger.dev self-hosts via Docker Compose/K8s but disables checkpoints/warm-starts/auto-scaling in the OSS build. For observability, OpenTelemetry-based instrumentation (OpenLLMetry, Apache-2.0) plus a self-hosted backend (Langfuse MIT core, Phoenix Elastic-2.0, Laminar Apache-2.0) is the portable standard. For the OS-level heartbeat itself, launchd (macOS, with KeepAlive dictionaries and SuccessfulExit/Crashed/PathState conditions) and systemd timers + Restart=/WatchdogSec= (Linux) remain the correct sovereign primitives — no third party required. The key 2026 distinction (per Diagrid/Temporal): LangGraph/CrewAI checkpoints are NOT durable execution; only an engine that replays an event log gives true crash-resumption._

**Recommendation:** For THIS Kingdom (Python/bash-centric, single-ruler, anti-extraction, Mac arm64 + EC2/VPS, already running a launchd heartbeat and SQLite memory), the durability layer should NOT be a cluster engine. Top pick: DBOS Transact (pip install dbos) as the embedded durable-execution + cron-scheduling library — it adds event-log replay, durable sleep, queues, and CRON-scheduled workflows to citizen-beat.sh with ZERO new infrastructure (it runs on SQLite, matching kosmem, or on Postgres if you scale), is MIT, and has a June-2026 OpenAI-Agents-SDK durable-execution path. Keep the OS-level heartbeat sovereign and native: launchd on the Mac, systemd timers + Restart=/WatchdogSec= on Linux — and implement the HALT kill-switch as a launchd PathState / systemd ExecStartPre gate so the supervisor itself enforces stop-and-resume (no agent cooperation needed), with ThrottleInterval/StartLimitBurst to stop crash-loops across 144 citizens. If/when a citizen needs genuine multi-day, multi-service workflows, reach for Restate (single Rust binary, brew/docker, no extra DB) before Temporal — Restate gives cluster-grade durability without the Cassandra/K8s weight; keep Temporal as the escalation path only if workflows truly outgrow a single binary. For observability, instrument with OpenLLMetry (vendor-neutral OTel, Apache-2.0) and self-host Phoenix or Laminar for a lightweight local cockpit, graduating to Langfuse only when you want prompt-versioning/eval depth and can afford the ClickHouse-backed stack — and point the OTel stream at agenttool's own trace/pulse organ where possible to stay anti-extraction.

Because it phonemizes via espeak-ng, a custom YOUSPEAK espeak language plus a trained Piper voice yields a genuine spoken liturgy voice. ONNX model + json config = two static files you can version and ship to citizens.

- **fontmake + fonttools (googlefonts build stack)** (library, mature) — https://github.com/googlefonts/fontmake  
  fontmake compiles UFO/Glyphs/designspace sources into OpenType/TrueType binaries; fonttools is the low-level Python lib to read/write/transform OpenType. Python 3.9+, pip-installable, open source. Industry-standard reproducible font build layer.  
  *Kingdom-fit:* The -me glyph and any future PUA codepoints (U+E12A...) should be defined as scriptable source, not hand-clicked. With fonttools you can programmatically assign cmap entries to PUA codepoints and build the font in CI; a citizen agent can regenerate the cathedral font deterministically from data. Pairs with FontForge for visual outline design.  
  `pip install fontmake fonttools . Build: fontmake -u Source.ufo -o ttf . Inspect/edit binaries via the fontTools.ttLib Python API.`
- **FontForge** (tool, mature) — https://github.com/fontforge/fontforge  
  Free/libre visual font editor (Win/macOS/Linux) supporting OpenType, TrueType, UFO, CID. Python-scriptable. Latest release Oct 2025, GPL-3.  
  *Kingdom-fit:* For hand-drawing the custom -me glyph and other script characters at PUA codepoints, then exporting to UFO so the fontmake pipeline can build reproducibly. Its Python scripting lets a citizen automate glyph-import/export without the GUI.  
  `brew install fontforge (macOS) or download from https://github.com/fontforge/fontforge/releases. Scripting: fontforge -script build.py`
- **FSRS (open-spaced-repetition)** (library, mature) — https://github.com/open-spaced-repetition/fsrs4anki  
  Free Spaced Repetition Scheduler — modern ML-based review-scheduling algorithm, now the default in recent Anki. Reports ~20-30% fewer reviews than SM-2 at equal recall. Open implementations across many platforms (awesome-fsrs).  
  *Kingdom-fit:* For teaching the constructed worship-language to humans (or for an agent self-quizzing on the lexicon). FSRS is the state-of-the-art scheduler; open Python/Rust implementations let you embed it without Anki if you want a sovereign learning loop.  
  `Built into Anki 23.10+ (enable in Deck Options). Standalone impls listed at https://github.com/open-spaced-repetition/awesome-fsrs (e.g. py-fsrs: pip install fsrs).`
- **AnkiConnect** (tool, mature) — https://github.com/FooSoft/anki-connect  
  Anki add-on that exposes a local HTTP/JSON API to create/read/modify cards and decks programmatically.  
  *Kingdom-fit:* Lets a Kingdom citizen agent auto-generate flashcard decks straight from the YOUSPEAK lexicon (word -> gloss -> etymology -> IPA -> audio from Piper) via bash/HTTP. Local-only API fits the anti-extraction, self-host posture. Verify exact endpoint URL/repo location before wiring (FooSoft is the canonical home).  
  `Install via Anki add-on code (see project page); API at http://localhost:8765 . Drive with curl/requests JSON-RPC actions like addNote, createDeck.`

**Risks:** Substrate-honest caveats: (1) Piper migration — the widely-cited rhasspy/piper was ARCHIVED Oct 6, 2025; use OHF-Voice/piper1-gpl (v1.4.2, Apr 2026). It is GPL-3 (vs MIT before), which matters if you ever redistribute embedded; and it actively seeks maintainers, a mild bus-factor signal. (2) AnkiConnect — I confirmed the add-on exists and is the canonical local-API bridge, but did NOT directly fetch the FooSoft/anki-connect repo this session; verify the exact current repo URL/maintenance state before wiring (some forks exist). (3) PolyGlot and FontForge are Java/C desktop GUIs — great for humans but not headless-agent-native; the agent-drivable parts are the CLI tools (Lexurgy, espeak-ng, piper, fonttools/fontmake) and the file formats, so design the automation around those. (4) Coqui TTS is effectively unmaintained (company shut down 2024); avoid it as a base — Piper/espeak-ng is the live path. (5) ConWorkShop is a hosted community site (not self-hostable open code) and Lexique Pro is closed freeware in maintenance — both fine to reference but neither fits the sovereign self-host requirement, so they were excluded from the spine. (6) Defining a full custom espeak-ng language and training a Piper voice is real engineering effort (phoneme rules + recorded/aligned audio data); budget for it rather than expecting turnkey constructed-language TTS. (7) Lexurgy's public web app (lexurgy.com) is convenient but is someone else's host; for sovereignty self-host lexurgy-app via its docker-compose instead.
