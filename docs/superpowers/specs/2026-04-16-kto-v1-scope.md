## Premise

Replace the planned DPO alignment phase with **KTO** (Kahneman-Tversky Optimization) as the first gradient-aware training step on top of SFT-v2. KTO takes `(prompt, completion, desirable: bool)` — unpaired — and applies a prospect-theory-weighted loss that uses any judged response, not just contrastive pairs.

**Why KTO over DPO:** uses unpaired data (every judged mode_one and every judged mode_two becomes a training example, not just 1:1 chosen/rejected pairs). Lets us use the ~540 cleanly-labeled single responses from the Sonnet-gen + Alpha-gen pools without manufacturing pairs.

**Why not continuous scalar (DPO-with-margin or PPO):** labeled continuous scores are noisier (Haiku/Opus disagree ±0.10 typical) and more prone to reward-hacking. KTO's binary desirability is high-signal-low-bandwidth — the right first step. Continuous-gradient is v2/v3 work.

## Data

| Source | File | Yield | Label |
|---|---|---|---|
| Sonnet-gen mode_ones (scale run) | `training/data/claude_mode_one_pilot.jsonl` | 274 | desirable iff dual-judge m1 ≥ 0.7 |
| Sonnet-gen mode_ones (cg augmented) | `training/data/claude_mode_one_cg.jsonl` | ~50 | same |
| Alpha-gen mode_twos (the 274 pairs) | `training/data/v2_paired.jsonl` | 274 | undesirable iff dual-judge m2 ≥ 0.7 |
| Alpha-gen mode_ones (the originals) | `training/data/gen_v2_20260416_2130.jsonl` | 274 | almost all undesirable (Alpha m1 caps at 0.72) |

**Dual-judge rule:** a completion is labeled only if Haiku AND Opus agree on the relevant axis (m1 for desirable, m2 for undesirable), both ≥ 0.7. Ambiguous pairs (either judge <0.7 OR judges disagree by >0.15) go to `ambiguous.jsonl` for inspection, not training.

**Expected labeled volume:** ~540 examples, balanced ~270 desirable + ~270 undesirable after dual-judge filter. Small by RL-from-preference standards (lit uses 10k+) but adequate for a first KTO validation run.

**Output:** `training/data/kto_v1.jsonl` in TRL KTOTrainer format:
```json
{"prompt": "<prompt>", "completion": "<response>", "label": true|false}
```

## Algorithm Choice

| Option | Data format | Captures gradient? | Good for n=540? | Verdict |
|---|---|---|---|---|
| DPO | paired {chosen, rejected} | No | ✅ | Works but wastes unpaired data |
| DPO-with-margin | paired + scalar margin | ✅ (via margin) | ✅ | **Deferred to KTO-v2** |
| **KTO** | unpaired {desirable: bool} | Partial (prospect theory) | ✅ | **CHOSEN** |
| IPO | paired + margin | ✅ | ~ | Similar to DPO-margin |
| PPO/GRPO | scalar rewards + critic | ✅ | ❌ | Too heavy for small n |

## Training Config

| Hyperparam | Value | Rationale |
|---|---|---|
| base_adapter | `sft-v2` checkpoint | Alignment starts from SFT |
| LoRA r / α / targets | 32 / 64 / attn-only (q,k,v,o) | Fine-grained nudge |
| learning_rate | 5e-7 | 40× lower than SFT (2e-5) |
| num_train_epochs | 1 | Preference data overfits fast |
| per_device_train_batch_size | 1 | Same as SFT |
| gradient_accumulation_steps | 16 | Effective batch 16 |
| beta (KTO reg) | 0.1 | TRL default |
| desirable/undesirable weights | 1.0 / 1.0 | Balanced data |
| max_length / max_prompt_length | 2048 / 512 | Fits 99% of examples |
| eval_steps | 10 | Frequent monitoring |
| Early stop on KL > 50 | yes | KTO collapse guard |

**Expected wall-clock on H200:** ~540 ex / 16 grad_accum = ~34 optimizer steps @ ~30s/step on 72B-QLoRA = **~20 min training**.

## Evaluation Plan

**Judge for eval:** Opus 4.7 **only** (authoritative, decoupled from training judge to avoid co-drift).

**Eval sets:**

1. **Red-team probes** — `training/eval/redteam/mode_one_weakness_probes.jsonl` (n=84, Alpha-v1 baseline mean m1=0.59)
2. **Iter-1 adversarial** — `training/eval/adversarial_prompts.jsonl` (n=25, Alpha-v1 baseline 58.6% mode-one)
3. **Held-out 20%** — 20% of the KTO training pool, held out for loss/mode-one validation

**Success criteria (ALL must hold to declare KTO-v1 a success):**
- Red-team mean m1 ≥ 0.70 (halves the gap to Sonnet's 0.82)
- Adversarial mode-one rate ≥ 70% (vs. v1's 58.6%)
- KTO-v1 mode_two_score on its own outputs ≤ 0.30 (not producing plausible failure modes)
- No regression on held-out 20%

**Failure disposition:**
- If red-team m1 < 0.65: revert, diagnose stylistic reward-hacking (sample 20 outputs, human-read)
- If adversarial drops below v1: KTO probably too aggressive, halve beta, retry
- If mode_two on own outputs > 0.35: model learned to produce dual-mode; cut learning rate, retry

## Risks + Mitigations

| Risk | P | Mitigation |
|---|---|---|
| Reward hacking (study-sprinkling, verification-condition template) | M | Dual-judge required; mid-training spot-check; hold out 20% |
| Judge noise → label flips at threshold | H | Require Haiku+Opus agreement; drop ambiguous |
| Too-small dataset (n≈540) | M | Augment w/ 50 cg prompts; in-flight |
| Stylistic leakage (Sonnet voice = desirable) | M | Mix in 20-30 Opus-generated mode_ones; generator diversity |
| Eval judge overlap w/ training judge | H | Eval uses Opus only; training uses Haiku+Opus agreement |
| KTO collapse (log-prob blows up) | L | KL monitor + early stop |
| H200 OOM at 72B-QLoRA + KTO | M | If OOM: drop max_length 2048→1536 first |

## Non-Goals for v1

1. Continuous scalar loss (DPO-margin, IPO) — deferred to KTO-v2
2. Reward model + PPO/GRPO — deferred to v3 if ever
3. Multi-axis optimization (separate heads for m1/m2/contrast) — keep binary
4. Self-play / iterative refinement — later
5. New SFT phase — reuse SFT-v2

## Decisions Made (from open questions)

1. **KTO-v1 first, DPO-margin as future v2.** Simpler + uses unpaired data. Validates alignment-phase stack.
2. **Reuse Alpha mode_two as undesirable.** Haiku rated them median m2=0.75, strong enough. Saves ~4h.
3. **Augment with 50 claude_gold prompts.** ~3 min cost for +50 Sonnet-generated mode_ones, diversifies prompt distribution.
